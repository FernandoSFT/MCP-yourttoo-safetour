# Roadmap - API Yourttoo

## yourttoo.com

https://www.yourttoo.com/apidocs

Examinar el swagger para detalles de los objetos de respuesta.

## Autenticación

POST /auth

- Función: Obtención de userid y accesstoken.
- Input: Email y password (Auth).
- Output: Token de acceso (loginResponse).
- Notas: Requerido antes de cualquier otra operación.

## Base de datos / Recursos

POST /find

- Función: Obtener inventario de países, ciudades, proveedores y tags.
- Input: Tipo (countries, cities, providers, tags) en payload (Find).
- Auth: Requiere accesstoken y userid.

## Descarga de Programas o Reservas

POST /fetch

- Función: Descargar detalle de un program o booking.
- Input: type y code (Fetch).
- Output: Objeto de programa (program).
- Auth: Sí.

## Búsqueda de Programas

POST /search

- Función: Buscar programas usando filtros (país, ciudad, tags, duración, precio, etc.).
- Input: Payload complejo (Search).
- Output: Resultados paginados (searchResponse).
- Auth: Sí.

## Reservar Programa

## POST /book

- Función: Crear reserva para un programa.
- Input: Detalles en formato BookRequest.
- Output: Detalles de reserva (Booking).
- Auth: Sí.

## Comprobar Disponibilidad

## POST /checkavailability

- Función: Verifica disponibilidad antes de reservar.
- Input: Igual a book (BookRequest).
- Output: Disponibilidad (AvailabilityResponse).
- Auth: Sí.

## Cancelar Reserva

POST /cancel

- Función: Cancela una reserva.
- Input: Datos de cancelación (CancelRequest).
- Output: Estado de reserva cancelada (Booking).
- Auth: Sí.

## Roadmap de Integración con la API de Yourttoo

## 1. Autenticación

Objetivo: Obtener el token o credenciales necesarias.

- Endpoint: /apiv2/autho similar (según especificado en Swagger)
- Método: POST

Resultado: loginResponse, estructura que contiene userid y accessToken que se añade en las cabeceras http del resto de endpoints del API para la autenticacion de las peticiones

```
{
    "userid": "string",
    "accessToken": "string"
}
```

## 2. Carga de Inventario Inicial (Meta-información)

Objetivo: Obtener los catálogos base para usar en filtros de búsqueda.

Consulta el siguiente endpoint:

- Payload: seleccionar el recurso a obtener: { “type”: “indicar tipo de recurso” }
- Endpoint: /apiv2/find
- Método: POST

| Recurso | Uso |
| --- | --- |
| /countries | Lista de países disponibles |
| /cities | Ciudades disponibles |
| /tags | Temas o características del programa |
| /providers | Agencias o proveedores turísticos |

Consejo: Almacenar estos datos en caché o base de datos para usarlos como sugerencias/filtros en un frontend o backend o para automatizar búsquedas iterativas.

## 3. Búsqueda de Programas

Objetivo: Buscar programas turísticos mediante filtros.

- Endpoint: /apiv2/search
- Método: POST

```
Estructura del payload (ejemplo):
{
    "filter": {
        "countries": ["tr"],
        "cities": ["antakya-tr"],
        "tags": ["short-city-breaks", "self-drive"],
        "providers": ["OMT341"],
        "pricemin": 100,
        "pricemax": 1000,
        "mindays": 3,
        "maxdays": 15,
        "maxresults": 100,
        "sort": "asc",
        "page": 0
    }
}
```

- Iteración:
- Paginación con page ( $0,1,2 \ldots$ hasta que la respuesta esté vacía).
- maxresults Limite: 100. (para nuestro entorno de test recomendado entre 10-20 resultados por pagina)
- Puedes recorrer combinaciones como:
- Todos los paises
- Todas las ciudades de un país.
- Todos los programas por proveedor.
- Combinación de múltiples filtros si es necesario.

## 4. Descarga de Catálogo Completo (modo “batch”)

Objetivo: Descargar todos los programas para guardarlos en tu sistema.

- Ejecutar búsquedas iterativas por país, ciudad o proveedor.
- Guardar cada programa con su:
- ID único
- Itinerario diario
- Descripciones
- Release
- Titulo
- Precio minimo,
- Pax minimos para operacion
- Incluido
- Categorias
- Política de cancelación
- Disponibilidad

## 5. Búsqueda en Tiempo Real (modo “live search”)

Objetivo: Consultar programas directamente desde la API sin guardarlos localmente.

- Igual que el punto 3, pero sin almacenar en base de datos.
- Asegúrate de aplicar filtros basados en selección del usuario (ej: país, días, ciudad).
- Útil para agencias que no desean almacenar datos y prefieren obtenerlos en el momento.

## 6. Búsqueda por Selección de Catálogo

## Modo alternativo (no documentado)

- Se realiza una selección previa de programas en la plataforma de Yourttoo (por backoffice o configuración). Esta tarea es manual y la gestion de este catalogo es consensuada y gestionada por yourttoo.
- Después, cualquier llamada al / search solo devuelve los programas de esa selección, sin respetar filtros.
- Se puede aplicar paginación, pero los filtros no tienen efecto.
- Ideal para mostrar un subconjunto específico de programas a un cliente/usuario.

## 7. Comprobación de Disponibilidad

Objetivo: Antes de reservar, confirmar que el programa está disponible.

- Endpoint: /apiv2/checkavailability
- Método: POST
- Parámetros: Codigo del programa + fecha concreta (“YYYY/MM/DD”) + acomodacion
- Respuesta:
- Detalles del programa para esa fecha
- Reserva provisional (estructura equivalente a una confirmada)

## 8. Reserva Final (si aplica)

Objetivo: Confirmar una reserva si la disponibilidad es positiva.

- Usualmente, se requiere enviar un payload igual o similar al de la comprobacion de disponibilidad. Consultar Swagger para estructura.
- Se almacena el ID de la reserva y se actualiza su estado.

## Recomendaciones Finales

1. Límites y paginación: no más de 100 elementos por página.
2. Token: refrescar o renovar si hay expiración.
3. Catálogo seleccionado: ideal para escenarios tipo “landing page” o campañas.
4. Modo batch vs live:
- Batch $=$ descarga periódica + guardado local.
- Live $=$ siempre desde la API, sin guardar nada.
